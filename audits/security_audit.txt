L'infrastructure réseau réelle du projet est la suivante (à vérifier dans
docker-compose.yml et nginx/default.conf avant de faire des hypothèses) :

Navigateur → Cloudflare (edge, WAF, TLS) → Cloudflare Tunnel (cloudflared, connexion
sortante uniquement, aucun port publié sur l'hôte) → Nginx (image officielle nginx:latest,
config montée en volume, TLS interne + en-têtes de sécurité) → Gunicorn/Flask

Il n'y a ni Box/Routeur avec port exposé, ni Nginx Proxy Manager : ce point a déjà fait
l'objet d'une correction (migration du 2026-07-05, commit 32dda22). Ne pars pas d'une
autre topologie sans l'avoir d'abord vérifiée dans le code.

Prends en compte cette architecture dans l'audit de sécurité, notamment pour :
- les headers HTTP (voir nginx/default.conf),
- la gestion des IP réelles (extensions.py::get_client_identifier, vérification CIDR),
- le nombre de sauts de proxy de confiance (PROXY_TRUSTED_HOP_COUNT, ProxyFix),
- Cloudflare et le tunnel sortant,
- TLS/HTTPS,
- les risques de bypass du reverse proxy,
- les limitations de débit (Flask-Limiter + Redis),
- les protections DDoS côté Cloudflare,
- la configuration Nginx réellement utilisée (pas de Dockerfile Nginx custom).

Le projet utilise deux modes de configuration :

  OMDB_API_KEY, EMAIL_APP_PASSWORD) + cloudflare_tunnel_token, lus depuis
  /run/secrets/<nom> (Docker Secrets), et flask/prod.env pour les variables non sensibles.
